# ADR-001: 密码加密选择 BCrypt

## 上下文（Context）

用户密码需要安全存储到 MySQL 数据库。需要选择一种加密方案，在安全性和性能之间取得平衡，同时考虑实现复杂度。

## 决策（Decision）

**选择 BCrypt（强度 10）作为密码加密方案。**

## 理由（Rationale）

1. **安全性**：BCrypt 是自适应哈希算法，可随硬件性能提升调整计算强度，抵抗暴力破解。
2. **内置盐值**：自动生成并存储盐值，无需额外维护盐值字段，减少存储复杂度。
3. **行业标准**：Spring Security 默认支持，社区成熟，文档丰富。
4. **性能可接受**：强度 10 下，单次加密耗时约 50ms，登录接口响应时间可控制在 200ms 内。

## 被拒绝的方案（Rejected Alternatives）

| 方案 | 拒绝理由 |
| :--- | :--- |
| **MD5** | 无盐值，彩虹表攻击风险极高；已被证明不安全 |
| **SHA-256** | 虽比 MD5 安全，但无盐值时仍存在字典攻击风险 |
| **PBKDF2** | 需手动管理盐值和迭代次数，实现复杂度高于 BCrypt |

## 影响（Consequences）

- **正向影响**：
  - 密码存储安全性达到行业标准。
  - 无需额外维护盐值字段，数据库设计简化。
- **负面影响**：
  - BCrypt 输出长度为 60 字符，需要确保数据库密码字段长度 ≥ 60。
  - 加密计算消耗一定 CPU 资源（约 50ms/次）。

## 后续演进（Future Considerations）

- 若未来硬件性能提升，可考虑将强度从 10 调整至 12。
- 需持续关注 NIST 关于密码存储的最新建议。
