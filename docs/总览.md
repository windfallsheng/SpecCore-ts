# SpecCore 总览

## 一、命令全景

### 🔵 日常 7 步流

```
init         项目初始化
word2spec    需求文档导入 (Word/MD → Spec)
analyze      需求分析 + 源码对标 + 架构影响
split        拆分任务 (11份文档 + IMPACT)
execute      执行开发 (模板生成代码)
pr           自动创建 Pull Request
done         一键收尾 (validate→archive→sync)
```

### 🟡 偶尔用 (16)

```
lifecycle    任务状态机 (dev→test→review→done)
rollback     回滚任务 + 删除分支 + 归档
merge-check  合并冲突预测
tracker      全量需求变更追踪
ops          操作历史查看
new-task     新建任务
bugfix       创建 Bug 修复任务
```

## 三、Spec → Code 理念

```
  "Code by Spec, Not by Vibe"

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  📝 Spec  │ → │  🔍 分析  │ → │  📦 任务  │ → │  💻 代码  │
  │ 需求规格  │    │ 对标源码  │    │ 11份文档  │    │ 模板生成  │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘
       ↑                                               │
       │              📜 CONSTITUTION                  │
       │          (强制规则 + 质量关卡)                  │
       └────────────────── 验证 ───────────────────────┘

  每个阶段都需用户确认（[y]保存 / [N]取消）
  分析 → 确认 → 拆分 → 确认 → 执行 → 关卡 → 完成
```

## 五、质量关卡

```
  in_progress ──→ testing ──→ review ──→ done
       │              │           │
       │         TEST.md      REVIEW.md
       │         完成？        审批？
       │              │           │
       └──── 关卡 ────┴─── 关卡 ──┘

  🚫 跳过 testing → review: 阻止
  🚫 跳过 review → done:    阻止
  ⚠️ TEST.md 未完成 → review: 阻止
  ⚠️ REVIEW.md 未审批 → done: 阻止
```

## 六、注解标记

```
  开发者只需在代码中加一行注释，analyze 自动精准读取：

  // @spec-module 订单管理    → 定义模块边界
  // @spec-api POST /api/xxx  → 标记接口归属
  // @spec-entity Order        → 标记数据实体
  // @spec-dependency 用户模块 → 声明模块依赖
```

## 八、架构设计评价

### 设计亮点 ✅

| 设计 | 评价 | 对比竞品 |
| :--- | :--- | :--- |
| Word→Spec→Code 全链路 | 独有。竞品都从手写 Spec 开始 | 无人做到 |
| 中文 NL 意图识别 (38种) | 独有。竞品均为英文生态 | 无人做到 |
| Git 分支 + 依赖自动检测 | 独有。竞品不管理代码仓库 | 无人做到 |
| 11 份文档/Task | 最丰富。Spec-Kit 8 份，其他更少 | 领先 |
| Constitution 自动检测 | 从项目文件推导，非手动编写 | 领先 Spec-Kit |
| 风险双维度评分 | 需求关键词 + 源码模式，含缓解建议 | 独有 |

### 待改进 ⚠️

| 问题 | 对比 | 影响 |
| :--- | :--- | :--- |
| 68 个命令 → 新人困惑 | Spec-Kit 仅 5 个 slash 命令 | 学习曲线 |
| 代码生成是模板，非 AI | Kiro/Tessl/BMAD 全用 AI | 复杂逻辑弱 |
| 缺乏 IDE 插件 | Kiro 是完整 IDE | 需要切终端 |
| 社区/生态为 0 | Spec-Kit 73K stars | 推广难度 |

### 结论

**SpecCore 是全流程覆盖最广的 SDD CLI 工具，没有之一。**
但在代码生成智能化（模板 vs AI）和易用性（命令数 vs slash commands）上需向竞品学习。
最佳策略：**做好 Spec 管理强项，代码生成交给 WorkBuddy/外部 AI**，各司其职。
