```
传统模式:  需求变更 → 直接改代码 → 文档滞后 → 知识丢失
SpecCore:     需求变更 → 先更新 Spec → 记录变更履历 → 再改代码
```

> ⌨️ `speccore xxx` = 终端 | 🤖 `/spec-xxx` = AI Slash | 💬 `"xxx"` = 对 AI 说话（三种入口，效果相同）

# SpecCore 场景实战指南

> 从真实开发场景出发，逐步演示 SpecCore 的完整使用流程。

## 场景一：从零启动新项目

```bash
# ⌨️ CLI 终端
cd my-project
speccore init
# 💬 或对 AI 说: "初始化一个 SpecCore 项目"

# 2. 导入已有代码到全量层
speccore import --project=backend --path=./src --type=backend
speccore import --project=frontend --path=./web --type=web
# 💬 "导入后台和前台项目"

# 3. 创建第一个期次
speccore iteration create --name=Q3 --from=2026-07 --to=2026-09
# → 期次-001-Q3（全局唯一编号自动分配）

# 4. 配置技术栈
speccore config --tech backend --set "Java 17, Spring Boot 3.2, JPA, MySQL 8.0"
speccore config --tech frontend --set "Vue 3, Element Plus, Vite"
```

**验证**:
```bash
ls .speccore/
# → CONSTITUTION.md  SETTINGS.md  GLOBAL/  PROJECT/  ITERATIONS/
# → .speccore/local/counters.json（全局唯一计数器）
```

## 场景二：需求文档导入

**背景**: PM 给了需求文档，有 Word 格式也有 Markdown 格式，不同端在不同文档中。

### 方式一：Word 文档转换导入

```bash
# 单文件导入
speccore word2spec -f 后台PRD.docx -i Q3 -p 后台管理端
# 💬 "把 Q3 的后台需求 Word 转成 Spec"

# 批量导入（推荐）—— 一次命令，全部搞定
speccore word2spec --files \
  "后台PRD.docx=后台管理端,后台H5接口.docx=后台H5端,Web需求.docx=Web,H5需求.docx=H5" \
  -i Q3
```

### 方式二：Markdown 文档直接导入

```bash
# .md 文件直接导入，无需 pandoc
speccore word2spec --files \
  "后台接口.md=后台管理端,前端页面.md=Web,小程序页面.md=H5" \
  -i Q3

# Word 和 MD 可以混着导入
speccore word2spec --files \
  "后台PRD.docx=后台管理端,前端页面.md=Web" \
  -i Q3
```

### 导入后的文件结构

| 文件 | 类型 | 说明 |
| :--- | :--- | :--- |
| `{端名}需求.md` | 手动维护 | 各端独立的需求文档 |
| `REQUIREMENT.md` | **自动生成** | 各端汇总，供 `iteration split` 使用 |
| `INDEX.md` | **自动生成** | 端列表索引，供 `split` 智能端检测 |
| `images/` | **自动生成** | Word 中提取的图片 |

### 需求变更时该改哪个？

| 变更来源 | 操作 |
| :--- | :--- |
| PM 给了新版本 Word | 重新执行 `word2spec --files`（自动覆盖 `{端}需求.md` + 刷新 `REQUIREMENT.md`） |
| 开发中微调需求 | 用 `speccore change --task=Task-001 --desc="xxx"`（不要手动改 REQUIREMENT.md） |
| 需要调整分端结构 | 改 `{端名}需求.md`，然后重新 `word2spec` |
| 全量层同步 | `REQUIREMENT.md` → `speccore sync-global --direction=to_global` |

> ⚠️ `REQUIREMENT.md` 和 `INDEX.md` 是衍生文件，下次 `word2spec` 会覆盖。**不要手动编辑这两个文件。**

**验证**:
```bash
ls 期次-Q3/00-需求文档/
# → 后台管理端需求.md 后台H5端需求.md  Web需求.md  H5需求.md  REQUIREMENT.md  INDEX.md  images/
```

## 场景十五：需求拆分全流程（全量层 → 期次 → 任务）

**方式一：从 Word/MD 文档导入后拆分**
```bash
# 1. 批量导入各端需求（自动合并到 REQUIREMENT.md）
speccore word2spec --files "admin.docx=后台管理端,h5.md=后台H5端,web.docx=Web" -i Q3

# 2. 预览拆分
speccore iteration split --dry-run

# 3. 执行拆分（自动检测端 + 按端分配）
speccore iteration split
# → Platforms: 后台管理端, 后台H5端, Web          ← 从 INDEX.md 自动检测
# → ## 后台管理端需求 下的 section  → 只建 backend/管理/
# → ## 后台H5端需求   下的 section  → 只建 backend/H5/
# → ## Web端需求       下的 section  → 只建 frontend/Web/

# 4. 手动指定端
speccore iteration split --platforms=web,mini,h5
```

**方式二：从全量需求池拉取**
```bash
# 1. 查看全量需求池
speccore global-status
# → REQ-001  用户认证   待规划

# 2. 拉取需求 → 生成期次
speccore iteration-from-global --reqs=REQ-001,REQ-002 --name=期次-2026-Q3

# 3. 执行拆分
speccore iteration split --iteration=期次-2026-Q3

# 4. 查看进度
speccore progress --iteration=期次-2026-Q3 --detail
```

## 场景二十四：多开发者并行

**背景**: 张三开发 Task-001，李四开发 Task-002，需要不冲突地并行工作。

**操作**:
```bash
# 张三
git checkout -b 260715-订单管理-zs
vim 期次-Q3/Task-001-订单管理/backend/REQ.md
speccore execute --task=Task-001 --force
speccore sync --task=Task-001 --detect

# 李四（同时）
git checkout -b 260715-用户认证-ls
vim 期次-Q3/Task-002-用户认证/backend/REQ.md
speccore execute --task=Task-002 --force
speccore sync --task=Task-002 --detect
```

**关键规则**:

| 规则 | 说明 |
| :--- | :--- |
| 不同 Task | 互不冲突，可完全并行 |
| 同一 Task | 不要两人同时改同一端（backend/frontend） |
| 全局配置 | 改 GLOBAL/* 或 CONSTITUTION 需相互告知 |
| PR 合并前 | 各自跑 `speccore sync --detect` 确认一致 |

**验证**:
```bash
# 检查不同分支各跑各的
speccore progress          # 整体进度
speccore health --task=Task-001  # 健康度
speccore health --task=Task-002
```

## 场景二十五：收尾归档

**背景**: Task 开发完成，PR 合并，需要走完整收尾流程。

**操作**:
```bash
# 1. 最后一致性检查
speccore sync --task=Task-001 --detect

# 2. 生成审计报告

# 4. 同步期次级需求到全局
speccore sync-global --iteration=Q3 --direction=to_global

# 5. 更新上下文（完成当前流程）
speccore current --task=Task-001 --status=done

# 6. 查看最终全景
speccore dashboard
    
# 🆕 或一键收尾
speccore done --task=Task-001
```

**对话式等价位**:
```
"收尾 Task-001"
"归档一下，同步到全局"
```

## 场景二十七：依赖任务分支管理

依赖任务开发时需要避免实体/组件重复生成。

**原理**：依赖方从被依赖方的分支分叉，自动继承已有代码。

```bash
# 1. 先查看依赖关系
cat 期次-001-Q3/IMPACT.md
# | Task-002: 订单查询 | → | Task-001: 用户管理 | `/api/users` |

# 2. 先执行无依赖的任务
speccore execute --task=Task-001 --force
# → 🌿 Created branch: feature/Task-001-用户管理

# 3. 依赖任务自动检测 base（推荐）
speccore execute --task=Task-002 --force
# → 🔗 检测到依赖 → 🎯 自动从 Task-001 分叉
# → 🌿 Created branch: feature/Task-002-订单查询 (from feature/Task-001-用户管理)

# 4. 或手动指定 base
speccore execute --task=Task-002 --force --base=feature/Task-001-用户管理
```

**结果**：Task-002 分支上已有 Task-001 的 UserController 等代码，不会重复生成。
合并时 Task-002 → main 只带来增量变更。

**默认 base 分支**：在 `.speccore/SETTINGS.md` 中配置：
```yaml
git:
  default_base: develop   # 默认从 develop 分叉（省略则从当前分支）
```
