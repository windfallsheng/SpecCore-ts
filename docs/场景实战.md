# SpecCore 场景实战指南

> 从真实开发场景出发，逐步演示 SpecCore 的完整使用流程。

---

## 场景一：从零启动新项目

```bash
# 1. 初始化
cd my-project
speccore init

# 2. 导入已有代码到全量层
speccore import --project=backend --path=./src --type=backend
speccore import --project=frontend --path=./web --type=web

# 3. 创建第一个期次
speccore iteration create --name=期次-2026-07 --goal="用户认证系统"

# 4. 创建任务
speccore new-task --name="用户登录" --platforms=web,h5 --type=feature
speccore new-task --name="用户注册" --platforms=web,h5 --type=feature
speccore new-task --name="密码重置" --platforms=web --type=feature

# 5. 查看进度
speccore progress --detail
```

---

## 场景二：日常开发执行

```bash
# 1. 查看任务计划和依赖
speccore plan --iteration=期次-2026-07

# 2. 选择一个任务开始开发（自动创建 Git 分支）
speccore execute --task=Task-001 --force
# → 🌿 Created branch: feature/Task-001-user-login

# 3. 查看当前分支关联
speccore current
speccore current --commit    # 生成 commit 消息

# 4. 在代码中打 @spec 标注
# // @spec Task-001-user-login
# @RestController
# public class AuthController { ... }

# 5. 反向同步（代码 → Spec）
speccore sync --dry-run       # 预览
speccore sync                 # 写入 TASK.md

# 6. 查看追溯链
speccore trace --task=Task-001

# 7. 标记完成
speccore update --task=Task-001 --status=completed

# 8. 归档
speccore archive --task=Task-001
```

## 场景三：Bug 修复流程

```bash
# 1. 创建 Bug 修复任务
speccore bugfix --name="登录超时" --desc="Token 过期导致 401"

# 2. 开始修复
speccore execute --task=Task-004 --force

# 3. 修复代码 + @spec 标注
# // @spec Task-004-login-timeout
# if (token.isExpired()) refreshToken();

# 4. 反向同步
speccore sync

# 5. 变更记录
speccore change --req=REQ-001 --desc="修复 token 过期逻辑" --task=Task-004          

# 6. 完成
speccore update --task=Task-004 --status=completed
```

## 场景四：多人协作

```bash
# 1. 架构师创建期次和任务

speccore new-task --name="用户登录" --platforms=web,h5
speccore new-task --name="用户注册" --platforms=web,h5

# 2. 后端开发领取任务

# 4. 查看团队进度
speccore progress --detail

# → 提示：该需求变更会影响 Task-001, Task-002

# 6. 生成交接文档
speccore handover --iteration=期次-2026-07
```

## 场景五：需求变更处理

```bash
# 1. 分析变更影响
speccore impact --req=REQ-001
# → 影响范围: Task-001(用户登录), Task-002(用户注册)

# 2. 创建基线（变更前快照）
speccore baseline --name=before-change

# 3. 描述变更
speccore change --req=REQ-001 --desc="登录增加短信验证码"

# 4. 更新受影响的 Task
speccore update --task=Task-001 --status=in_progress

# 5. 对比变更前后
speccore diff --source=before-change --target=current

# 6. 验证
speccore validate --iteration=期次-2026-07
```

## 场景六：长任务链分批执行

```bash
# 有 12 个 Task，每 3 个一批
speccore execute --all --batch-size=3 --dry-run    # 预览分批方案
speccore execute --all --batch-size=3 --force      # 执行

# 执行到第 2 批中断了...
# → 从第 3 批继续
speccore execute --resume

# 交互式：手动选择哪些任务执行
speccore execute --all --interactive
```

## 场景七：CI/CD 集成

```yaml
# .github/workflows/speccore-ci.yml
name: SpecCore Validate

on: [push, pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install SpecCore
        run: npm install -g speccore
      - name: Validate Specs
        run: speccore validate --full
      - name: Check Health
        run: speccore health
      - name: Progress Report
        run: speccore progress --format=json > progress.json
```

```bash
# 本地安装 Git hooks
speccore hooks install
# → pre-commit: 检查 @spec 标注
# → pre-push: 运行 speccore validate
```

## 场景十四：需求拆分全流程（全量层 → 期次 → 任务）

```bash
# → REQ-001  用户认证   待规划

speccore iteration split --iteration=期次-2026-Q3 --dry-run
```

```bash
# 1. 单任务归档（移出工作目录，保留记录）

# 2. 整个期次已完成的全部归档
speccore archive --all --iteration=期次-2026-07
```

## 场景十六：开发中途需求变更

```bash
# 1. 变更前打基线（出问题可回滚）
speccore baseline --name=before-login-change

# 2. 分析影响范围
speccore impact --req=REQ-001
# → 影响: Task-001(登录), Task-003(权限)

# 3. 记录变更
speccore change --req=REQ-001 --desc="登录增加短信验证码" --task=Task-001

# 4. 手动更新 Spec 文件
#    编辑 Task-001/backend/REQ.md
#    编辑 Task-001/_shared/API_CONTRACT.yaml

# 5. 更新状态
speccore update --task=Task-001 --status=in_progress

# 6. 重新同步
speccore sync-global --iteration=期次-2026-07

# 7. 对比差异
speccore diff --source=before-login-change --target=期次-2026-07

# 8. 验证
speccore validate
```

### 整个期次需要大变更

```bash
# 1. 先打基线
speccore baseline --name=before-refactor

# 2. 逐个 Task 走变更流程（impact → change → 改Spec → validate）

# 3. 如果期次要废弃重来
speccore delete --iteration=期次-2026-07-old --force
speccore iteration-from-global --reqs=REQ-001,REQ-002 --name=期次-2026-07-new
```

**口诀：** `baseline 保底 → impact 看影响 → change 留记录 → 改 Spec → validate 验证`

## 场景十七：Spec 校验报错 — 怎么修

```bash
# 1. 检查当前项目
speccore validate
# → ❌ Task-001/backend/REQ.md: 缺少 "## 3. 接口定义" 章节
# → ❌ Task-002/_shared/API_CONTRACT.yaml: YAML 语法错误 第15行

# 2. 指定任务检查
speccore validate --task=Task-001

# 3. 自动修复（补全缺失章节）
speccore validate --task=Task-001 --fix

# 4. JSON 格式输出（CI 用）
speccore validate --format=json

# 5. 安装 Git hooks（push 前自动校验）
speccore hooks install
```

**常见错误：**
- REQ.md 缺章节（用 `--fix` 自动补）
- API_CONTRACT.yaml 缩进不一致（手动对齐 2 空格）
- PROJECT_GRAPH.md 表格格式错（不要手动改，用命令）

## 场景十八：平台管理常见误区

### 忘了多端

```bash
# 创建时只选了 web，后来发现需要 h5
speccore new-task --name="用户登录" --platforms=web --type=feature

# ❌ 不要删了重建，直接加平台
speccore platform-add --name=h5 --tech="H5 Mobile"
# 然后手动编辑 Task-001/_shared/TASK.md，h5 的职责部分
```

### 平台搞多了

```bash
# 创建时选了 web,h5,miniapp 三端，实际只有 web
speccore new-task --name="管理后台" --platforms=web,h5,miniapp --type=feature

# 只执行 web 端即可，其他端不会报错只会跳过
speccore execute --platform=web
speccore progress --platform=web
```

### 加新平台到项目

```bash
speccore platform-add --name=tablet --description="平板端" --tech="React Native"
# 后续 new-task 就可以选 tablet 了
speccore new-task --name="xxx" --platforms=web,tablet
```

## 场景十九：文件操作错误 — 手动改了不该改的

### 手动改了 PROJECT_GRAPH.md

```bash
# ❌ 直接编辑了 PROJECT_GRAPH.md 的状态表格
# 表现：下次 execute/plan 运行时旧数据覆盖了你的修改

# ✅ 正确做法
speccore update --task=Task-001 --status=completed
speccore plan --iteration=期次-2026-07
```

### 手动改了 context.json

```bash
# ❌ 直接编辑 .speccore/local/context.json 的 currentTask
# 表现：下次命令运行读取到的值和预期不一致

# ✅ 正确做法
speccore execute --task=Task-003  # 自动更新 currentTask
```

### 手动删了 Task 文件夹

```bash
# ❌ rm -rf 期次-2026-07/Task-005
# 表现：INDEX.md 残留孤儿引用，context 计数不准

# ✅ 修复
speccore index-update             # 清理索引
speccore validate                 # 检查一致性

# ✅ 正确做法
speccore delete --task=Task-005   # 安全删除
```

## 场景二十二：文件保存即校验 — `speccore watch`

### 对比：手动 vs 自动

| | 手动 `validate` | `speccore watch` |
| :--- | :--- | :--- |
| **反馈时机** | 写完后跑命令 | 每次保存就反馈 |
| **发现错误** | 事后（可能忘了） | 当场（秒级） |
| **YAML 缩进错误** | `--fix` 批量修 | 第 1 次保存就提示 Tab 问题 |
| **REQ 章节缺失** | 集中报 3-4 个错 | 缺失 1 个就提示 |
| **使用方式** | 每次敲命令 | 开终端挂着，无人值守 |

### 使用流程

```bash
# 终端 1：开启 watch
speccore watch
# → 👀 Watching Spec files... (Ctrl+C to stop)

# 终端 2：编辑 Spec
vim Task-001/backend/REQ.md

# 每次 Ctrl+S 保存 → 终端 1 即时反馈：
#   ✅ Task-001/backend/REQ.md
#   ⚠️ Task-001/_shared/API_CONTRACT.yaml (tab indentation — use 2 spaces)
#   ⚠️ Task-001/backend/REQ.md (missing: 接口定义, 验收标准)

# 只监听特定期次
speccore watch --iteration=期次-2026-07
```

### 不用 watch 的流程（对比）

```bash
# 编辑 Spec... Ctrl+S
# 编辑 Spec... Ctrl+S
# 编辑 Spec... Ctrl+S
# （全程无反馈，不知道有没有问题）

# 最后一次性跑
speccore validate
# → ❌ 3 个错误 → 逐个修改 → 再跑 validate → 再改...
```
