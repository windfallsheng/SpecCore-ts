# SpecCore 文件操作安全指南

> 哪些文件可以手动改？哪些必须用命令？一篇文章讲清楚。

---

## 一、可以直接手动改的（安全区）

本质是"描述性文字"，不参与逻辑计算，不构成依赖关系。

| 文件 | 可手动改的内容 | 为什么安全 |
| :--- | :--- | :--- |
| `REQ.md` | 需求描述、背景、业务规则文字 | AI 开发时读取这些文字作为指令，不参与结构化解析 |
| `TECH.md` | 技术方案描述、设计思路 | 同上，是给 AI 和开发者看的"说明书" |
| `GLOSSARY.md` | 术语定义 | 纯文本索引，修改后执行 `speccore sync --reverse` 即可更新索引 |
| `*.md` 中的注释行 | 备注、TODO、注意事项 | 不影响任何流程 |

改完后执行 `speccore sync --reverse --task=xxx` 把文字"吸"回 JSON 数据库。

---

## 二、不建议手动改的（高风险区）

本质是"元数据/状态/关系"，是系统运行逻辑的依赖。

| 文件 | 内容 | 手动改的后果 |
| :--- | :--- | :--- |
| `PROJECT_GRAPH.md` 表格 | 状态、负责人、依赖关系、工期 | 下次 CLI 执行 `plan/execute` 时，渲染逻辑直接覆盖表格，你的修改会丢失 |
| `.task-type` 文件 | 任务类型（feature/bugfix/...） | 类型决定生成什么模板、执行什么检查，改错会影响整个生命周期 |
| `API_CONTRACT.yaml` | 接口路径、参数、返回值结构 | 改错一个缩进，`speccore validate` 报错；前后端契约全靠它，手动改极易导致校验失败 |
| `.speccore/data/*.json` | 所有结构化数据（任务、期次元数据） | 改错一个引号或拼错枚举值（如 completed 写成 compleed），`speccore plan` 直接报 ZodError 卡死 |

这几类必须用命令改：

```bash
speccore change Task-001 "修改描述"            # 变更需求
speccore plan --assign=张三                     # 调整负责人
speccore execute --task=Task-001 --force        # 执行任务
```

---

## 三、可以手动改但必须立即验证的（慎改区）

本质是"写给人看的文档生成源"，修改后需要同步回系统。

| 文件 | 内容 | 正确流程 |
| :--- | :--- | :--- |
| `GLOBAL/REQUIREMENT.md` | 全量需求描述（Markdown 正文） | 改完后立即执行 `speccore sync-global`，把新需求同步到索引 |
| `CHANGELOG.md` | 变更历史记录 | 手动追加没问题，但建议让 `speccore change` 自动追加 |

**原则：改完就同步，不让人工修改成为"游离态数据"。**

---

## 四、总结：一句话判断

| 类别 | 判断标准 | 做法 |
| :--- | :--- | :--- |
| ✅ 可以手动改 | 文件内容是"给人读的文字"——描述、备注、说明、指南 | 直接改，改完 `speccore sync --reverse` 回写数据库 |
| ❌ 不要手动改 | 文件内容是"给机器读的结构"——YAML 缩进、JSON 字段、表格状态列、依赖关系 | 用对应的 `speccore` 命令修改 |
| ⚠️ 可以手动改，但必须立即同步 | 全量需求文档、全局索引 | 改完立即执行 `speccore sync-global` |

---

## 五、实用速查表

| 你想做什么 | 正确方式 | 错误方式 |
| :--- | :--- | :--- |
| 改需求描述文字 | 打开 `REQ.md` 改，然后 `speccore sync --reverse` | 直接改 .json 文件 |
| 把任务状态改为"已完成" | `speccore change Task-001 "状态更新"` | 改 `PROJECT_GRAPH.md` 表格 |
| 增加一个新接口定义 | 手动编辑 `API_CONTRACT.yaml`，然后 `speccore validate` 校验 | 改完直接提交，不校验 |
| 调整某个任务的负责人 | `speccore plan --assign=张三` | 改 `PROJECT_GRAPH.md` 表格 |
| 在全量需求里加一个新需求 | 打开 `GLOBAL/REQUIREMENT.md` 改，然后 `speccore sync-global` | 只改文件不执行同步 |
| 改技术方案描述 | 打开 `TECH.md` 直接改 | — |
| 迁移 Shell 版到 CLI 版 | `speccore migrate` | 手动复制粘贴 |
| 重建需求索引 | `speccore index-update` | 手动编辑 `INDEX.md` |

---

## 六、核心理念

CLI 不禁止你手动改任何文件，但 CLI 会在执行命令时用结构化数据覆盖掉"元数据/状态/关系"类的手动修改。

描述性文字（需求描述、技术方案、注释）永远安全，因为它们是"给人读的"，CLI 不会用 JSON 去覆盖这些文字。

如果你想让手动修改生效，用 `speccore sync --reverse` 把你的修改"吸"回数据库。

**手动改文字 → 自由高效；命令改结构 → 安全可靠。两者各司其职。**
