# SpecCore 完整使用指南

> 两种方式：💬 自然语言对话 或 ⌨️ CLI 命令，效果完全一致。

---

## 一、两种使用方式

| 方式 | 怎么用 | 适合场景 |
| :--- | :--- | :--- |
| **💬 对话 AI** | 自然语言说需求，AI 直接读写 Spec | 配置规则、改需求、查看状态 |
| **⌨️ CLI 命令** | 用命令行执行操作 | 批量执行、CI/CD、脚本自动化 |

### 💬 对话式（不需要命令）

你说的话，AI 直接操作对应 Spec 文件：

```
💬 "把 CONSTITUTION.md 里异常类改成 ApiException"
   → AI 自动找到 .speccore/CONSTITUTION.md 并修改

💬 "我们在前端用 React + Ant Design"
   → AI 自动更新 .speccore/GLOBAL/TECH_STACK.md

💬 "Task-001 要加一个批量删除"
   → AI 更新 REQ.md，然后教你重新 execute

💬 "现在的开发进度怎么样了"
   → AI 读取 PROJECT_GRAPH.md，回复进度

💬 "宪法里面有什么代码规则"
   → AI 读取 CONSTITUTION.md 的 spec-rule 区块回复
```

**不需要记忆文件路径，也不需要 `/spec` 前缀。**

### ⌨️ 命令行式

```bash
speccore execute --task=Task-001 --force        # 执行任务
speccore sync --task=Task-001 --detect           # 检测差异
speccore change --task=Task-001 --desc="..."     # 需求变更
speccore progress                                # 查看进度
speccore validate                                # 验证完整性
speccore pattern -n crud -t Task-001             # 保存模式
```

## 二、场景对照表

| 你想做什么 | 💬 AI 对话 | ⌨️ CLI 命令 |
| :--- | :--- | :--- |
| 查看规则 | "有哪些代码规范" | `cat .speccore/CONSTITUTION.md` |
| 配置规范 | "异常统一用 ApiException" | `speccore config --rule exception-handler --set "..."` |
| 配置技术栈 | "我们用 Java 17 + JPA" | `speccore config --tech backend --set "Java 17, JPA"` |
| **导入 Word 需求** | **"帮我把 Q3 的 PRD 转成 Spec"** | **`speccore word2spec -f prd.docx -i Q3 -p 后台`** |
| 创建期次 | — | `speccore iteration create --name=Q3` |
| 拆分任务 | — | `speccore iteration split` |
| 开发任务 | "开发 Task-001" | `speccore execute --task=Task-001 --force` |
| 检查差异 | "代码和 Spec 一致吗" | `speccore sync --task=Task-001 --detect` |
| 需求变更 | "Task-001 要加批量删除" | `speccore change --task=Task-001 --desc="新增: 批量删除"` |
| 查看进度 | "进度怎么样了" | `speccore progress` |
| 保存模式 | "把 CRUD 存为模式" | `speccore pattern -n crud -t Task-001 -d "CRUD"` |
| 紧急修复 | "线上紧急 bug，先跳过同步" | `speccore execute --task=Task-001 --hotfix --force` |
| 验证完整性 | "检查一下 Spec 有没有问题" | `speccore validate` |
| 同步 Spec | — | `speccore sync --reverse --task=Task-001` |
| 查看依赖 | "Task-001 依赖哪些任务" | `speccore trace --task=Task-001` |
| 回滚变更 | "刚才改的不对，帮我回滚" | `speccore rollback --task=Task-001 --confirm` |
| 查看备份 | "有哪些备份文件" | `speccore rollback --task=Task-001 --list` |

> ⚙️ **前置依赖**：`speccore word2spec` 需要 pandoc 进行 Word → Markdown 转换。首次运行时若未安装，命令会提示安装（`brew install pandoc`）或建议使用 AI 对话中的 `word2md` 技能。

## 三、完整操作步骤

### 新期次从零开始

```bash
# 1. 初始化 + 建期次
speccore init
speccore iteration create --name=Q3

# 2. 导入 Word/MD 需求文档（批量，推荐）
speccore word2spec --files "后台PRD.docx=后台管理端,Web需求.docx=Web,H5需求.docx=H5" -i Q3
# 或单文件: speccore word2spec -f 后台PRD.docx -i Q3 -p 后台管理端

# 3. 拆分任务（自动检测端 + 按端分配）

# 5. 同步到全局
speccore sync-global --direction=to_global

# 5. 补每个 Task 的接口表格，然后开发
vim 期次-Q3/Task-001-订单管理/backend/REQ.md
speccore execute --task=Task-001 --force

# 6. 开发完成后检查
speccore sync --task=Task-001 --detect
speccore validate
```

### 💬 对话式等价位

```
"创建 Q3 期次"
"把这两个 Word 需求文档转成 Spec，放到 Q3 的后台和 Web"
"拆一下任务"
"同步到全局"
"开发 Task-001"
"检查一下有没有不一致"
```

## 四、项目配置（init 后一次配好）

`speccore init` 生成三个配置文件，填入团队信息即可：

### TEAM.md — 成员 + 分支命名

```markdown
格式: {YYYYMMDD}-{任务名}-{缩写}
成员: 张三 | zs | zs@ex.com
```

### CODE_INDEX.md — 工程路径

```markdown
| backend | src/main/java/com/example | Spring Boot |
```

### CONSTITUTION.md — 分支规则

```markdown
<!-- spec-rule: git-branch -->
分支格式: YYYYMMDD-任务名-缩写
<!-- /spec-rule -->
```

> 配置后说"开发 Task-001"→ AI 自动 `git checkout -b 260721-订单管理-zs`。

## 七、AI 修改 Spec 的自动保障

当你在对话中说"Task-001 要加批量删除"，AI 执行**两阶段确认**：

### 第一阶段：变更分析

```
AI:  📋 变更分析:
     文件: REQ.md
     变更: + DELETE /api/v1/orders/batch — 批量删除
     影响: Task-002 任务分配（引用了订单接口，需回归）
     风险: 🟢 低（新增接口，不破坏现有）
     确认继续？
```

### 第二阶段：执行计划

```
用户: 确认
AI:  📝 执行计划:
     1. 更新 REQ.md → 新增 DELETE /api/v1/orders/batch
     2. 追加变更履历 (REQ.md 末尾)
     3. 标注影响: Task-002
     开始执行？

用户: 确认
AI:  ✅ 已完成
     - REQ.md 已更新，变更履历已追加
     - 受影响: Task-002 任务分配
     - 下一步: speccore execute --task=Task-001 --force
     - 验证: speccore sync --task=Task-001 --detect
```

这些规则写在 CONSTITUTION.md 的「AI 操作规则」中，对话中的 AI 自动遵守。

## 八、核心理念

```
💬 对话:  说需求 → AI 写 Spec
⌨️ 命令:  执行 Spec → 生成代码 → 检测差异
```

CLI 不禁止你手动改任何文件。描述性文字（需求、方案、注释）永远安全。
