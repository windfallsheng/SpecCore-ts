# SpecCore 文件操作安全指南

> 哪些文件可以手动改？哪些必须用命令？改完要做什么？一篇文章全覆盖。

---

## 一、三层文件总览

### .speccore/ 层（全局配置，`init` 时创建）

| 文件 | 来源 | 能手动改？ | 改了之后 |
| :--- | :--- | :--- | :--- |
| `CONSTITUTION.md` | init 自动生成 | ✅ 随便改 | 下次 execute 读入上下文 |
| `SETTINGS.md` | init 自动生成 | ✅ 随便改 | 下次 execute 读入上下文 |
| `POST_COMPLETION.md` | init 自动生成 | ✅ 随便改 | 下次 execute 读入上下文 |
| `config/platforms.yaml` | init 生成 + platform-add 追加 | ⚠️ 用 `config --set` | new-task 依赖它 |
| `local/context.json` | 命令自动维护 | ❌ 别碰 | 命令自动覆盖 |
| `local/execution-state.json` | execute 自动生成 | ❌ 别碰 | 断点续传依赖 |
| `.git-mapping.json` | execute/current 自动写 | ❌ 别碰 | 分支映射依赖 |
| `GLOBAL/INDEX.md` | index-update 生成 | ❌ 用 `index-update` | 手动改会被覆盖 |
| `GLOBAL/REQUIREMENT.md` | 手动写 + sync-global 同步 | ✅ 手动追加 | 写完 `sync-global` |
| `GLOBAL/PROJECTS/<name>/` | import 自动填充 | ✅ 改需求描述 | 写完 `sync-global` |
| `GLOBAL/BASELINES/` | baseline 命令创建 | ❌ 别碰 | 用 baseline 命令管理 |

### 期次层（`期次-xxx/`，iteration 命令创建）

| 文件 | 来源 | 能手动改？ | 改了之后 |
| :--- | :--- | :--- | :--- |
| `PROJECT_GRAPH.md` | iteration split 生成 | ❌ **别碰** | 用 `update` / `plan` 改；手动改会被 CLI 覆盖 |

### Task 层（`Task-xxx/`，new-task 创建）

| 文件 | 来源 | 能手动改？ | 改了之后 |
| :--- | :--- | :--- | :--- |
| `backend/REQ.md` | new-task 生成骨架 | ✅ 随便写需求 | 写完 `speccore sync` |
| `backend/TECH.md` | new-task 生成骨架 | ✅ 随便写方案 | 无后续操作 |
| `backend/TASK.md` | new-task 生成 + sync 填充 | ✅ 补充细节 | 写完 `speccore sync` |
| `_shared/API_CONTRACT.yaml` | new-task 生成骨架 | ⚠️ YAML 敏感 | 写完 `speccore validate` |
| `_shared/TRACE.md` | sync 自动生成 | ❌ 别碰 | sync 会覆盖 |

## 二、可以直接手动改的（安全区）

本质是"描述性文字"，不参与逻辑计算，不构成依赖关系。

| 文件 | 可手动改的内容 | 为什么安全 |
| :--- | :--- | :--- |
| `REQ.md` | 需求描述、背景、业务规则 | AI 读取这些文字作为指令，不参与结构化解析 |
| `TECH.md` | 技术方案、设计思路 | 给 AI 和开发者看的"说明书" |
| `GLOSSARY.md` | 术语定义 | 纯文本索引 |
| `*.md` 中的注释 | 备注、TODO、注意事项 | 不影响任何流程 |
| `CONSTITUTION.md` | 技术规范、编码标准 | execute 时读取，不改数据结构 |
| `SETTINGS.md` | 框架配置描述 | 同上 |

---

## 三、不建议手动改的（高风险区）

本质是"元数据/状态/关系"，是系统运行逻辑的依赖。

| 文件 | 内容 | 手动改的后果 |
| :--- | :--- | :--- |
| `PROJECT_GRAPH.md` 表格 | 状态、负责人、依赖关系 | CLI 执行 `plan/execute` 时覆盖表格 |
| `.task-type` 文件 | 任务类型（feature/bugfix/...） | 类型决定生成什么模板 |
| `API_CONTRACT.yaml` | 接口路径、参数、返回值 | 缩进错误 → validate 报错 |
| `*.json` 数据文件 | 结构化数据 | 拼错枚举值（如 completed → compleed）→ ZodError |
| `INDEX.md` | 需求索引 | index-update 会覆盖 |
| `TRACE.md` | 追溯链 | sync 会覆盖 |

必须用命令改：

```bash
speccore change --req=REQ-001 --desc="修改描述"
speccore update --task=Task-001 --status=completed
speccore execute --task=Task-001 --force
```

## 四、慎改区（改完立即同步）

| 文件 | 正确流程 |
| :--- | :--- |
| `GLOBAL/REQUIREMENT.md` | 改完 → `speccore sync-global` |
| `CHANGELOG.md` | 建议用 `speccore change` 自动追加 |
| `API_CONTRACT.yaml` | 改完 → `speccore validate` |
| `config/platforms.yaml` | 用 `speccore config --set`，或 `platform-add` |

## 五、速查表

| 你想做什么 | ✅ 正确方式 | ❌ 错误方式 |
| :--- | :--- | :--- |
| 改需求描述 | 编辑 `REQ.md` → `speccore sync` | 直接改 .json |
| 标记任务完成 | `speccore update --task=T-001 --status=completed` | 改 PROJECT_GRAPH.md 表格 |
| 新增接口 | 编辑 `API_CONTRACT.yaml` → `speccore validate` | 改完不校验 |
| 调整负责人 | `speccore update --task=T-001 --assignee=张三` | 改 PROJECT_GRAPH.md |
| 加全量需求 | 编辑 `GLOBAL/REQUIREMENT.md` → `speccore sync-global` | 只改文件不同步 |
| 改技术方案 | 直接编辑 `TECH.md` | — |
| 重建索引 | `speccore index-update` | 手动改 INDEX.md |
| 删任务 | `speccore delete --task=T-001` | rm -rf |
| 改名 | `speccore rename --target=old --new-name=new` | mv + 手动改所有引用 |

---

## 六、核心理念

```
手动改文字 → 自由高效；命令改结构 → 安全可靠。
改完文字 → sync 回写；改完 Yaml → validate 校验。
```

CLI 不禁止你手动改任何文件，但 CLI 会在执行命令时用结构化数据覆盖"元数据/状态/关系"类的手动修改。描述性文字（需求、方案、注释）永远安全。
